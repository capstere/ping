Du har helt rätt: Instrument Error för MTB+INVALID triggas inte av “Invalid w/o error (SPC Ct=0)” (den delen är redan byggd för att bli Minor Functional) — utan av att ObservedCall blir UNKNOWN i Get-ObservedCallDetailed.

Exakt trigger i din zip (RuleEngine.ps1)

I Get-ObservedCallDetailed finns detta:
	•	För MTB görs bara tre overrides:
	•	MTB TRACE DETECTED ⇒ POS
	•	MTB DETECTED ⇒ POS
	•	MTB NOT DETECTED ⇒ NEG
	•	All “generic fallback” för tokens (INVALID, NO RESULT, ERROR) körs bara om assay inte är MTB:

if (-not ($ass -match '(?i)MTB')) {
    ...
    $hasErr = ($u -match '\bINVALID\b') -or ...
}
...
if (-not ($ass -match '(?i)MTB')) {
    if ($hasErr) { return Call='ERROR' }
    ...
}
return Call='UNKNOWN'

Så för MTB+INVALID:
	•	ingen MTB-override träffar
	•	inga RuleBank-patterns kanske träffar
	•	generic fallback blockeras
	•	⇒ return UNKNOWN
	•	⇒ Classify-Deviation ger UNKNOWN
	•	⇒ QC Summary-mappar UNKNOWN ⇒ Instrument Error / Okänt

Det är därför det händer.

⸻

Fix med “reducera istället för addera”

Vi ska inte lägga på nya MTB-specialfall, utan bara låta INVALID/NO RESULT/ERROR-token alltid bli ERROR även för MTB – utan att aktivera POS/NEG-tokenlogik för MTB (för då får vi “MIXED” pga “RIF Resistance NOT DETECTED”).
