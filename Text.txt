Ja — jag har öppnat zippen och hittat exakt varför MTB hanterar INVALID “annorlunda” i QC Summary.

Varför blir INVALID “Instrument Error / Okänd” för MTB?

I Modules/RuleEngine.ps1 finns funktionen Get-ObservedCallDetailed som bestämmer ObservedCall (POS/NEG/ERROR/UNKNOWN).

För icke-MTB kör skriptet en “generic fallback” som säger:
	•	om Test Result innehåller INVALID / NO RESULT / ERROR ⇒ ObservedCall = ERROR

Men för MTB var den fallbacken uttryckligen avstängd via villkor:

if (-not ($ass -match '(?i)MTB')) { ... generic fallback ... }

Alltså: MTB-rader med Test Result = INVALID hamnar ofta i ObservedCall = UNKNOWN (om inget mönster i RuleBank fångar den), vilket gör att QC Summary-switch klassar UNKNOWN som:
	•	Instrument Error
	•	Error Code = Okänt

Samtidigt kan Get-ErrorInfo ändå sätta ErrorName = “Invalid w/o error (SPC Ct=0)” (från RuleBank’s NamedBlanks), vilket gör att du ser just kombon du beskrev: Instrument Error + Okänd men med “Invalid w/o error (SPC Ct=0)” i Error Type-kolumnen.

Det är hela buggen.
